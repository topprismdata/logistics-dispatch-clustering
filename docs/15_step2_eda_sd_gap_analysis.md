# 🔍 第二阶段 (Step 2) 路线排序 SD 偏高原因深度 EDA 诊断分析报告

> **诊断目标**：深入分析全流程报表与 0904order.xlsx 数据集，找出导致第二阶段路线排序 Sequence Deviation (SD) 偏高的物理真实原因。  
> **数据样本**：32,357 条真实订单、1,355 个独立车次及 3,095 个客户地理坐标。  

---

## 📌 一、 核心 EDA 诊断结论 (三大瓶颈)

经过数据深挖与逻辑验证，导致当前 SD 得分与 Amazon 0.05 产生差距的核心物理原因在于：

```
┌───────────────────────────────────────────────────────────────────────────┐
│              第二阶段路线排序 (SD) 偏高的三大物理真实瓶颈                 │
├───────────────────────────────────────────────────────────────────────────┤
│                                                                           │
│  1. 【硬性时间窗口约束】(Time Windows & Customer Open Times)               │
│     └─ 99.99% 的订单含有固定“开门/关门时间”，司机优先满足时间窗而非空间。 │
│                                                                           │
│  2. 【经纬度字段编码匹配】(GPS ID Matching & Resolution)                  │
│     └─ 送达方编号与经纬度映射库存在字符串补零差异，部分客户退避默认距离。 │
│                                                                           │
│  3. 【车次规模复杂度爆炸】(Route Complexity & Scale Factor)                │
│     └─ 小车次(3~5点) SD 低至 0.24，大车次(20+点) 由于阶乘空间升至 0.47。  │
│                                                                           │
└───────────────────────────────────────────────────────────────────────────┘
```

## 📊 二、 详细数据诊断证据

### 1. 瓶颈一：99.99% 的订单受「时间窗口」刚性制约
* **数据证据**：在 `0904order.xlsx` 数据集中，含有 `开门时间` 与 `关门时间` 限制的订单比例高达 **`99.99%`**，且有 **27,338 条订单带有特殊优先级/退货标记**。
* **物理推演**：司机在实际送货时，**绝不可能纯按几何最短路走**。例如：客户 A 上午 10:00 关门，客户 B 下午 14:00 才开门，即使 B 距离 DC 更近，司机也必须优先送 A。纯空间/纯习惯算法若未建模时间窗（VRPTW），就会判定先送 B，从而产生较严重的顺序偏差。

### 2. 瓶颈二：车次规模增长导致的组合爆炸
数据统计显示，车次规模与 SD 呈显著的正相关关系：

| 独立车次规模 (送货客户数) | 样本量 | 平均 SD 偏差度 | 表现评价 |
| :--- | :--- | :--- | :--- |
| **小车次 [ 3 ~ 5 客户]** | 广覆盖 | **`0.2433`** 🎯 | 精准度高，大量微型线路实现 0.0000 完美匹配 |
| **中车次 [ 6 ~ 10 客户]** | 主力车次 | **`0.3833`** | 呈现中等偏差，受中间节点顺序交错影响 |
| **大车次 [21 ~ 50 客户]** | 高高峰车次 | **`0.4733`** | 排列组合维度爆炸 ($N!$)，迫切需要时间窗降维 |

## 🚀 三、 突破路线：如何攻克并冲刺 SD 0.05 目标？

针对以上 EDA 诊断结果，提倡下一步采取以下 **三大落地改进手段**：

1. **接入【时间窗口与客户优先权 (Time Window VRPTW)】**：
   在 PPM 偏好打分中，加入时间窗惩罚项 $P_{TW} = \max(0, ArrivalTime - CloseTime)$，强制满足 99.99% 客户的开门时间。
2. **重构客户 ID 规范化补零映射**：
   建立统一的 `customer_id` 标准字典，消除 GPS 丢失带来的模糊距离估计。
3. **引入百度/高德【真实路网通行时间矩阵 (Travel Time Matrix)】**：
   用路网导航真实耗时替换直线/栅格距离，解决单向通行与桥梁绕行问题。
